Консольное приложение на .net, предназначенное для мягкого, но надёжного удаления дубликатов файлов после их побайтового сравнения, с сохранением всех логов.

Приложение проверялось в среде Ubuntu 24.04 или MacOS 26.
В разработке ТЗ для валидации использованы ChatGPT 5.2 Thinking, Gemini 3 Pro, Claude Opus 4.6, каждый их них внёс свою лепту

Настраивается следующим образом:
1) В конфиге указывается адрес до master-директории: это директория, которая останется после выполнения приложения. Если она пустая - то запрашивается сразу после старта приложения
2) Также в конфиге указывается адрес до slave-директории: это директория, которую нужно сравнить с master. Если она пустая - то запрашивается после старта приложения
3) Количество одновременных потоков сканирования - по умолчанию 4

Работает приложение следующим образом:
1) После запуска запрашивает отсутствующие директории. 
2) Далее стартует первый этап работы.
    - Если одна из директорий является поддиректорией другой - приложение завершается с ошибкой.
    - Если в таблице уже содержатся строки, то первый этап работы (сканирование) пропускается.
3) Запускает два параллельных (асинхронных) процесса: сканирование master-директории и сканирование slave-директории.
4) Каждый из процессов сначала получает список всех файлов в директории и во всех поддиректориях, и делит их на количество одновременных потоков
5) Затем стартуют одновременные потоки сканирования, которые параллельно получают строгие хеш-суммы для каждого из файлов и сохраняет в таблицу files следующую информацию:
    - fileId - guid
    - master или salve
    - полный путь до файла
    - размер файла в байтах
    - LastWriteTimeUtc
    - CreationTimeUtc
    - строгая хеш-сумма
    - статус файла (Discovered → Hashed → CandidateMatched → ByteVerified → MovedToSlaveQuarantine / MergedToMaster / MovedToMasterQuarantine / DifferentContent / Error)
6) Прогресс сканирования (число обработанных/общее число) отображается в консоли, с визуальным отображением прогресс-бара
7) После завершения сканирования запускается второй этап работы, он выполняется в одном потоке для избежания проблем с race condition и подобными вещаями
8) Делается salve left join master по size и строгой хеш-сумме.
    - В результате этого left join образуется view (master_slave_join), содержащая пары записей slave, для которых есть master и записей slave, для которых нет master
9) Если в master_slave_join есть записи, рядом с директорией slave создаётся директория slave_quarantine
10) Для каждой строки в master_slave_join проводится проверка. 
11) Если файл из slave побайтно совпадает с файлом в master - он переносится в директорию slave_qarantine, причём с сохранением относительного пути до файла
    - Перенос осуществляется по правилам "Безопасного переноса", описанным ниже
    - В таблице files отмечается, что файл из slave перенесён в slave_qarantine
12) Если для файла в slave нет аналога в master, то он переносится в директорию master, с сохранением относительного пути до файла
    - Перенос осуществляется по правилам "Безопасного переноса", описанным ниже
    - В таблице files добавляется добавляется master-запись, а slave-запись отмечается как merged с указанием id master-записи
    - Новая master-запись содержит LastWriteTimeUtc, CreationTimeUtc и строгую хеш-сумма для перенесённого файла, а не копию из slave-записи
13) Если для файла в slave побайтное сравнение с файлом master выявило различие - он остаётся в оригинальной директории, а slave-запись в таблице отмечается отдельным признаком, со ссылкой на master-файл, для которого побайтное сравнение нашло разницу.
    - В консоль при этом выводится предупреждение, содержащее путь до slave-файла
14) В процессе работы прогресс (число обработанных/общее число) отображается в консоли, с визуальным отображением прогресс-бара
15) Далее стартует третий этап работы - поиск дубликатов в master-директории. Он также работает в одном потоке для избежания проблем при работе 
16) Выполняется join по всем master-записям между собой по полю строгой хеш-суммы, с сортировкой по пути до файла в алфавитном порядке
    - в итоге должны получить view с дублями (duplicates), содержащую записи OriginalFilePath, OriginalHash, DuplicateFilePath, DuplicateHash, идентификаторы OriginalId, DuplicateId
    - view duplicates строится так, чтобы каждая пара была один раз: условие OriginalId < DuplicateId и явно исключается OriginalFilePath == DuplicateFilePath.
17) Если во view duplicates находятся строки, то рядом с директорией master создаётся директория master_quarantine
18) Для каждой строки в duplicates проводится проверка:
    - Сначала проверяется, что файл OriginalFilePath существует и не был перемещен (проверка по БД). Если Original перемещен, пара пропускается.
    - Если original-файл побайтно совпадает с duplicate-файлом, duplicate-файл переносится в директорию master_quarantine, причём с сохранением относительного пути до файла
    - Перенос осуществляется по правилам "Безопасного переноса", описанным ниже
    - В таблице files отмечается, что duplicate-файл из master перенесён в master_quarantine
19) Если для original-файла побайтное сравнение с duplicate-файлом выявило разницу, тогда об этом выводится предупреждение в консоль, duplicate-файл в таблице отмечается как побайтно-несовпадающий со ссылкой на OriginalFileId
20) После завершения работы в slave-директории проводится чистка директорий, которая удаляет все директории, не содержащие данных
21) В процессе работы прогресс (число обработанных/общее число) отображается в консоли, с визуальным отображением прогресс-бара

Данные хранятся и обрабатываются в базе данных SQLite, которая создается в директории приложения.
База данных содержит таблицы для хранения информации о файлах, их хешах и результатах сравнения.
Приложение использует Entity Framework Core для работы с базой данных.

"Безопасный перенос" файлов осуществляется максимально надёжным способом:
1. Перечитывается запись из базы для проверки того не перенесён ли уже файл
2. Проверяется, что файл не изменился: проверяются size и LastWriteTimeUtc в таблице и у файла.
3. Если что-то отличается - пересчитывается хеш у файла
   - если он отличается от хеша в таблице, то файл не переносится, в таблице он отмечается как Error
   - если он сопадает - продлжаем перенос
4. Если size и LastWriteTimeUtc совпадают, то файл переносится (доверяем hash в таблице)
5. Создаётся целевая директория
6. Файл копируется во временное имя в целевой директории
7. Файл атомарно переименовывается в оригинальное имя в целевой директории (или в fileName_1.txt, fileName_2.txt и т.д. в случае конфликта)
8. В базу данных записывается факт переноса и новое имя файла, если к имени был добавлен номер
9. Только после этого файл удаляется из оригинальной директории
