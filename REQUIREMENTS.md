# DiDeduplicator - Requirements Specification

## Обзор

Консольное приложение на .NET 9 для мягкого и надёжного удаления дубликатов файлов. Сравнивает slave-директорию с master-директорией, перемещает дубликаты в карантин (не удаляет), уникальные файлы мержит в master. Побайтовая верификация перед любым перемещением. Полное логирование всех операций.

---

## Принятые решения

| Решение | Выбор |
|---|---|
| .NET версия | .NET 9 |
| Хеш-алгоритм | SHA-256 |
| Логирование | Serilog (файл + консоль) |
| Конфигурация | appsettings.json |
| Symlinks / недоступные файлы | Пропуск с записью в лог |
| Multi-match (slave → несколько master) | Побайтово проверять все до первого совпадения |
| Resume этапов 2 и 3 | Да, по статусам в БД |
| Итоговый отчёт | Подробный |
| Пустые файлы (0 байт) | Пропуск в JOIN, но сканируются и записываются в БД |
| Dry-run | Нет |

## Конфигурация (appsettings.json)

```json
{
  "DiDeduplicator": {
    "MasterDirectory": "",
    "SlaveDirectory": "",
    "ScanThreads": 4
  }
}
```

- **MasterDirectory**: путь до master-директории (если пуст — запрашивается при старте)
- **SlaveDirectory**: путь до slave-директории (если пуст — запрашивается при старте)
- **ScanThreads**: количество одновременных потоков сканирования (по умолчанию 4)

## База данных

**Хранилище**: SQLite, файл создаётся в директории приложения.
**ORM**: Entity Framework Core (с провайдером SQLite).

### Таблица `Files`

| Поле | Тип | Описание |
|---|---|---|
| FileId | GUID (PK) | Уникальный идентификатор записи |
| Source | string (enum) | `Master` или `Slave` |
| FullPath | string | Полный путь до файла |
| FileSize | long | Размер файла в байтах |
| LastWriteTimeUtc | DateTime | UTC время последнего изменения |
| CreationTimeUtc | DateTime | UTC время создания |
| HashSha256 | string | SHA-256 хеш-сумма (hex) |
| Status | string (enum) | Статус файла (см. ниже) |
| QuarantinePath | string? | Путь после переноса в карантин (если применимо) |
| MergedToMasterFileId | GUID? | Ссылка на master-запись (для merged slave-записей) |
| ByteMismatchFileId | GUID? | Ссылка на файл, с которым обнаружено побайтное несовпадение |
| ProcessedAt | DateTime? | Время обработки записи |

### Статусы файлов

```
Discovered → Hashed → CandidateMatched → ByteVerified → [финальный статус]
```

Финальные статусы:
- **MovedToSlaveQuarantine** — slave-дубликат перемещён в slave_quarantine
- **MergedToMaster** — уникальный slave-файл перемещён в master
- **MovedToMasterQuarantine** — master-дубликат перемещён в master_quarantine
- **DifferentContent** — побайтное сравнение выявило расхождение (хеш совпал, но содержимое — нет)
- **Error** — ошибка при обработке (хеш изменился, файл недоступен и т.д.)
- **Skipped** — файл пропущен (symlink, недоступен, 0 байт)

## Этап 1: Сканирование

### Предусловия
- Проверка: ни одна из директорий не является поддиректорией другой (иначе — завершение с ошибкой)
- Если в таблице `Files` уже есть записи — этап пропускается целиком

### Процесс
1. Запускаются два параллельных (асинхронных) процесса: сканирование master и slave
2. Каждый процесс:
   - Получает список всех файлов в директории и поддиректориях
   - Symlinks, junction points и недоступные файлы — пропускаются с записью в лог и в БД (статус `Skipped`)
   - Делит файлы на `ScanThreads` частей
   - Запускает параллельные потоки хеширования (SHA-256)
   - Для каждого файла создаёт запись в `Files` со статусом `Discovered`, затем обновляет до `Hashed` после вычисления хеша
3. Пустые файлы (0 байт): сканируются и записываются в БД (статус `Skipped`), исключаются из последующих JOIN
4. Прогресс отображается в консоли: прогресс-бар с числом обработанных/общее

### Параллельная запись в SQLite
- Использовать WAL mode для SQLite
- Записи батчами (batch insert/update) для снижения lock contention

---

## Этап 2: Сравнение slave с master

### Предусловия
- Resume: пропускаются записи, у которых статус уже финальный (не `Hashed`)

### Процесс (однопоточный)

1. **LEFT JOIN**: slave LEFT JOIN master по (`FileSize`, `HashSha256`), исключая файлы со статусом `Skipped`
   - Результат: view `master_slave_join` — пары (slave_record, master_record или null)

2. **Для каждой записи в master_slave_join**:

   **Случай A: Есть совпадение с master (один или несколько)**
   - Побайтово сравнивать slave-файл с каждым master-кандидатом, пока не найдётся первое полное совпадение
   - Если совпадение найдено → перенести slave-файл в `slave_quarantine` (Безопасный перенос), статус `MovedToSlaveQuarantine`
   - Если ни один master-кандидат не совпал побайтово → slave-файл остаётся на месте, статус `DifferentContent`, `ByteMismatchFileId` = ID последнего проверенного master-файла, предупреждение в консоль

   **Случай B: Нет совпадения в master**
   - Перенести slave-файл в master-директорию (Безопасный перенос), с сохранением относительного пути
   - Создать новую master-запись в `Files` с фактическими `LastWriteTimeUtc`, `CreationTimeUtc` и `HashSha256` перенесённого файла
   - Slave-запись: статус `MergedToMaster`, `MergedToMasterFileId` = ID новой master-записи

3. **Карантин**: директория `slave_quarantine` создаётся рядом со slave-директорией, только если есть что переносить

4. Прогресс-бар в консоли

---

## Этап 3: Дедупликация внутри master

### Предусловия
- Resume: пропускаются записи, у которых статус уже финальный

### Процесс (однопоточный)

1. **Self-JOIN**: все master-записи между собой по `HashSha256` (исключая `Skipped`)
   - View `duplicates`: (`OriginalId`, `OriginalFilePath`, `OriginalHash`, `DuplicateId`, `DuplicateFilePath`, `DuplicateHash`)
   - Условие: `OriginalId < DuplicateId` AND `OriginalFilePath != DuplicateFilePath`
   - Сортировка по пути (алфавитная) — файл с меньшим путём считается Original

2. **Для каждой пары**:
   - Проверить, что Original не перемещён (статус в БД). Если перемещён — пропустить пару
   - Побайтовое сравнение Original и Duplicate
   - Если совпадение → перенести Duplicate в `master_quarantine` (Безопасный перенос), статус `MovedToMasterQuarantine`
   - Если расхождение → предупреждение в консоль, Duplicate: статус `DifferentContent`, `ByteMismatchFileId` = OriginalId

3. **Карантин**: директория `master_quarantine` создаётся рядом с master-директорией, только если есть что переносить

4. Прогресс-бар в консоли

---

## Этап 4: Очистка пустых директорий

После завершения этапов 2 и 3 — удаление всех пустых директорий в slave-директории (рекурсивно, снизу вверх).

## Безопасный перенос

Алгоритм переноса файла из `sourcePath` в `targetDir`:

1. Перечитать запись из БД — проверить, что файл ещё не перенесён (статус не финальный)
2. Сверить `FileSize` и `LastWriteTimeUtc` у файла на диске с записью в БД
3. **Если отличаются**: пересчитать SHA-256 файла
   - Хеш отличается от записи в БД → файл не переносится, статус `Error`, лог ошибки
   - Хеш совпадает → продолжаем перенос
4. **Если совпадают**: доверяем хешу в таблице, продолжаем перенос
5. Создать целевую директорию (с сохранением относительного пути)
6. Скопировать файл во временное имя в целевой директории (`.tmp_` + GUID)
7. Атомарно переименовать во финальное имя:
   - Исходное имя файла
   - При конфликте: `fileName_1.ext`, `fileName_2.ext`, ... (инкрементальный суффикс)
8. Записать в БД: факт переноса, `QuarantinePath` (включая новое имя, если добавлен суффикс)
9. Удалить файл из исходной директории
10. При любой ошибке на шагах 5–9: попытка очистки временного файла, статус `Error`, лог ошибки

## Логирование (Serilog)

### Назначение
- **Файл**: структурированные логи, файл рядом с БД (`logs/dideduplicator-{Date}.log`), ротация по размеру (100 MB)
- **Консоль**: краткие сообщения (прогресс, предупреждения, ошибки)

### Уровни
- **Information**: старт/завершение этапов, ключевые операции
- **Warning**: побайтное несовпадение, пропуск symlinks, недоступные файлы, конфликт имён при переносе
- **Error**: ошибки хеширования, ошибки переноса, изменившиеся файлы

## Нефункциональные требования

1. **Безопасность данных**: ни один файл не удаляется — только перемещается в карантин. Побайтовая верификация перед каждым перемещением.
2. **Идемпотентность**: повторный запуск безопасен — уже обработанные записи пропускаются по статусам.
3. **Производительность**: параллельное хеширование на этапе 1, потоковое чтение файлов для побайтового сравнения (без загрузки целиком в память).
4. **Надёжность**: WAL mode для SQLite, транзакции для критических операций, безопасный перенос с атомарным переименованием.
5. **Наблюдаемость**: структурированные логи через Serilog, прогресс-бары, подробный итоговый отчёт.
